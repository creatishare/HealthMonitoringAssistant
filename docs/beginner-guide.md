# 肾健康助手项目完全指南（写给非技术人员）

你好！这是一份专门为你准备的项目说明书。不涉及高深的专业术语，目的是让你对**“我们在做什么”**、**“代码是怎么组织的”**以及**“各个功能是怎么运转的”**有一个清晰、直观的认识。

---

## 1. 我们的项目是什么？

我们可以把**“肾健康助手”**想象成一个数字化的“私人健康大管家”，主要由两个部分组成：

- **前端（也就是用户的手机或浏览器看到的界面）**：相当于管家的**“脸”和“手”**，负责向用户展示图表、表单，接收用户的点击和输入。
- **后端（藏在服务器里的逻辑中心）**：相当于管家的**“大脑”和“笔记本”**，负责计算数据、判断指标是否异常、把数据妥善存进档案库（数据库）里，并在每天特定的时间提醒用户吃药。

---

## 2. 文件夹里都有什么？（代码地图）

当你打开项目的文件夹，你会看到很多英文命名的文件夹，不要怕，我们来一一对应：

```text
HealthMonitoringAssistant/ (你的项目大门)
├── src/                (最重要的源码仓库)
│   ├── frontend/       (前端：界面的代码都放在这里)
│   └── backend/        (后端：大脑的逻辑代码都放在这里)
│
├── docs/               (文档柜：放着产品需求、设计规范和这篇指南)
├── memory/             (AI的记忆库：记录了AI历次工作的历史和教训)
├── infrastructure/     (基建包：如果我们要把大管家搬到云服务器上，需要用到的安装图纸)
└── start-dev.sh        (启动按钮：双击或运行它，就能一键唤醒管家的前后端)
```

## 3. 前端：用户看到的界面是怎么做出来的？

进入 `src/frontend/` 文件夹，这里就像是一个 UI 组件的乐高积木工厂。

### 3.1 核心房间：`src/pages/`
这是我们各个页面的“蓝图”。
- `Dashboard.tsx`：**首页（仪表盘）**。你登录后看到的第一眼：打招呼、今日打卡（体重、血压、尿量）、即将要吃的药、各种化验指标的折线图，都是这个文件画出来的。
- `Medications.tsx` 和 `MedicationForm.tsx`：**用药管理**。前者是列出所有药的清单页，后者是添加和修改具体药物（比如设定每天几点吃）的填表页。
- `RecordForm.tsx` 和 `RecordDetail.tsx`：**健康记录（化验单）**。在这里手动输入你的肌酐、尿酸、他克莫司浓度，以及查看某一天录入的具体详情。
- `OCRUpload.tsx`：**智能识别功能**。用户在这里上传化验单照片，然后把提取出来的指标展示出来供用户核对。

### 3.2 帮手房间：`src/services/api.ts`
这里是**“通讯员”**。前端没有能力自己存数据，当你在页面上点击“保存”时，这个文件会把你的数据打包好，通过网络发送给“后端的大脑”。

## 4. 后端：大管家的大脑是怎么工作的？

进入 `src/backend/` 文件夹，这里决定了业务的流转。

### 4.1 档案柜管理员：`prisma/schema.prisma`
这是整个项目的**“账本骨架”**。如果你想知道我们到底给每个用户存了哪些数据，看这个文件就够了。
- 它定义了：`HealthRecord`（健康化验单，存着肌酐、尿酸、他克莫司浓度等）、`Medication`（每种药的名字、规格）、`Alert`（报警记录，比如浓度超标提醒）等等。

### 4.2 办事大厅：`src/controllers/` 和 `src/routes/`
这是管家的**“接线员”**。
- `routes`（路由）就像总机，当前端说“我要存一张化验单”时，路由会根据网址把它分发给对应的 `controller`（控制器）。
- `controller` 收到请求后，会先检查“这个人填没填日期？”如果填了，就把它交给服务部门去办理。

### 4.3 核心业务部门：`src/services/`
这是大管家的**“业务骨干”**。
- `health-record.service.ts`：负责把化验指标写进账本，并且画出折线图（把过去30天的数据搜集起来）。
- `ocr.service.ts`：这是**“翻译官”**。负责把图片发给百度的 AI 去识别文字，然后再一行一行地寻找类似“他克莫司 8.5 ng/mL”这样的字眼，提取出来给用户。
- `dashboard.service.ts`：负责在用户每天打开首页时，把体重、血压、尿量以及要吃的药，**汇总**成一个礼包送到前端去。

### 4.4 定时闹钟：`src/workers/`
这个文件就像一个**“闹钟管理员”**。它每隔一段时间就会去翻一下用药本，看看是不是有人快到吃药时间了（或者已经过了30分钟还没吃），如果是，它就会生成一条系统提示发给用户。

## 5. 当你“拍照识别化验单”时，背后发生了什么？

让我们串联一下前后端，看看一个真实的功能是怎么跑通的：

1. **你（前端）**：在手机上点击了“拍照识别”，拍了一张医院单子。
2. **通讯员（前端 API）**：带着图片敲开了后端的门。
3. **接线员（后端 Route/Controller）**：检查图片没问题，交给了翻译官（OCR 服务）。
4. **翻译官（OCR Service）**：将图片发给百度，百度返回了一堆纯文字。翻译官用我们写的规则（寻找关键词如肌酐、他克莫司）提取出精准的数字。
5. **通讯员（前端 API）**：把提取到的数字带回给你。
6. **你（前端）**：在页面上看到数字都填好了，检查无误，点击“保存”。
7. **业务骨干（后端 HealthRecord 服务）**：把这一天的化验数据永远保存在了**档案柜（数据库）**中。
