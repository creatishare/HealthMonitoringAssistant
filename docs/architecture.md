### 1.2 选型理由详解

#### 为什么选择 PostgreSQL 而非 MySQL？

1. **更好的 JSON 支持**: 便于存储 OCR 原始结果和灵活的化验单数据
2. **更严格的类型系统**: 减少数据类型错误，适合医疗数据的精确性要求
3. **更好的扩展性**: 支持数组类型、范围类型，适合存储参考值范围等数据
4. **成熟的地理信息支持**: 为未来扩展（如附近医院推荐）预留空间

#### 为什么选择 JWT 而非 Session？

1. **便于移动端适配**: 无状态设计，适合 PWA 和可能的未来移动端 App
2. **无状态便于扩展**: 不依赖服务器内存，方便水平扩展
3. **使用 HttpOnly Cookie 存储**: 兼顾安全性，防止 XSS 攻击
4. **支持刷新令牌**: 实现长期登录体验，适合老年用户

#### 为什么选择 Bull Queue 实现用药提醒？

1. **精确到分钟的定时任务**: 支持在指定时间触发提醒
2. **可靠的队列机制**: 基于 Redis，任务不会丢失
3. **支持重复提醒**: 未确认服药时可配置重复提醒（每30分钟，最多3次）
4. **任务状态追踪**: 可查询任务执行状态，便于监控

## 5. 部署架构

### 5.1 Docker Compose 部署

```yaml
# docker-compose.yml 概要
version: '3.8'
services:
  nginx:
    image: nginx:1.24-alpine
    ports:
      - "80:80"
      - "443:443"

  frontend:
    build: ./src/frontend

  backend:
    build: ./src/backend
    environment:
      - DATABASE_URL=postgresql://...
      - REDIS_URL=redis://redis:6379

  worker:
    build: ./src/backend
    command: npm run worker

  postgres:
    image: postgres:14-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
    volumes:
      - redis_data:/data
```

### 8.1 技术决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
